# Support Bot для @GPT4Telegrambot

Telegram-бот поддержки по модели **"бот сначала, Иван при необходимости"** без собственной БД.

## Что реализовано

- Главное меню с 5 сценариями:
  - Возврат / перенос
  - Купил не тот пакет
  - Оплатил, но не подключилось
  - Вопрос по пакетам
  - Передать Ивану
- Сбор минимальных данных по кейсу (включая `telegram_user_id` из `from.id`).
- Генерация карточки тикета в приватный саппорт-чат.
- Кнопки для оператора в карточке:
  - ✅ Закрыть
  - ↩️ Нужны данные
  - 👤 Взять Ивану
  - 🔁 Перенос
  - 💸 Возврат
- Команды:
  - `/myid`
  - `/case <номер>`
  - `/help`
- SLA watchdog (in-memory): просроченные кейсы эскалируются в `#escalated`.

## Важные ограничения (без БД)

- Состояние кейсов хранится **в памяти процесса** и в сообщениях Telegram (карточках в саппорт-чате).
- После рестарта процесса оперативный словарь кейсов очищается; критичные решения должны фиксироваться в карточке.
- Для полноценного восстановления после рестарта потребуется отдельный persistent-слой (или MTProto-история чата).

## Переменные окружения

- `BOT_TOKEN` — токен бота
- `SUPPORT_CHAT_ID` — ID приватного саппорт-чата
- `SUPPORT_AGENT_ID` — (опционально) Telegram ID Ивана для ограничения staff-кнопок
- `SLA_MINUTES` — (опционально, по умолчанию `30`) SLA в минутах
- `STARS_API_TOKEN` — (опционально) токен для авто-сверки Telegram Stars

## Запуск

```bash
python -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install -e .
python bot.py
```

## Примечание по Stars

В `PaymentService` встроен каркас для логики:

1. Проверить успешную оплату по пользователю/времени.
2. Если Stars начислены, но не списаны — списать.
3. Если списаны, но доступ не активирован — выдать пакет повторно (идемпотентно).

Сейчас это реализовано как шаблон-обработчик (`reconcile`) для дальнейшей интеграции с вашим платежным контуром.
